# Catalog SKU Obfuscation - Phased Plan

A lightweight catalog and special-order layer for Bellas XV that lets staff work with real designer SKUs while customers only ever see Bellas public codes and customer-safe product labels.

## Goal

Staff should type, search, sell, reorder, and report using the real internal SKU. Customer-facing surfaces should automatically render an obfuscated Bellas code instead. The staff workflow stays natural; the system handles the translation at the boundary.

The important boundary is not "admin users vs customer users." It is "staff surfaces vs customer documents." Admin invoice builder, reports, reorder views, and internal order logs show the real SKU. Customer invoice PDFs, portal pages, emails, SMS, receipts, and signed links show only the public code and a safe description.

## Decisions To Lock

- **Store both identifiers on the catalog row.** `internal_sku` is the real designer/style/color identifier staff uses. `public_code` is generated by the system and immutable once created.
- **Public codes are not manually typed.** They are minted on insert, unique, and never edited by staff.
- **Customer-facing renderers never read `internal_sku`.** PDFs, portal pages, receipt templates, public emails, and public SMS all use customer-safe helpers.
- **Invoice line descriptions cannot leak.** Catalog-backed lines should not render a staff-entered free-text `description` to customers. Use catalog-derived customer copy or a separate `public_description`.
- **Staff notes are staff-only.** Any internal free text belongs in `internal_notes`, never in a field that can hit a customer PDF.
- **Designer names are a deliberate policy choice.** Showing "Mori Lee" may help sales but can make the dress searchable. The default customer formatter should use a generic label or Bellas house name unless the business intentionally allows designer names on public documents.
- **Use named helpers, not a string audience switch.** `customerSku(item)` and `staffSku(item)` are harder to misuse than `displaySku(item, "customer")`.
- **Non-catalog charges ride on invoice lines.** `catalog_item_id` should be nullable so alterations, rush fees, deposits, discounts, and custom service lines do not require phantom catalog rows.
- **Public codes are minted by extending the existing `numbering_state` singleton.** Add a `catalog_public_code_seq INTEGER NOT NULL DEFAULT 0` column (no year column; catalog codes do not reset annually) and allocate via the same `SELECT ... FOR UPDATE` row lock that this repo already uses for invoice, quote, and payment numbering in `services/invoice_service.py`. Format as `BVX-{seq:05d}`. Do not introduce a separate Postgres sequence or a new state table.
- **Quotes have their own line-item write path.** This repo currently has a separate `quote_line_items` table and `QuoteEditor` independent from `invoice_line_items`. The catalog schema additions, write rules, search, and customer rendering must apply to both. Default assumption is two parallel write paths; only collapse to one if a deliberate refactor lands first.
- **Legacy invoice and quote lines are partially grandfathered.** Existing lines without a `catalog_item_id` keep rendering their `description` as-is on customer surfaces, because that text is already on issued PDFs and portal pages in customers' hands. Their `notes` column stops rendering to customers at the Phase 4 render swap, on every line including legacy: Phase 0 confirmed `notes` was never customer-safe by design, and grandfathering it would assume staff intended every typed note as customer-facing. The catalog path applies to new lines only. No backfill of historic invoices or quotes.
- **The legacy `description` and `notes` columns are write-frozen for new lines.** New catalog-backed lines do not write to either column; new non-catalog lines write only to `public_description` and `internal_notes`. Both columns stay in the schema for historic rows: `description` keeps rendering to customers on legacy lines (already on issued PDFs), `notes` does not (Phase 0 confirmed it was never customer-safe). The render swap stops `notes` exposure for all lines, including legacy.
- **`public_description` is for non-catalog lines only.** For catalog-backed lines, customer copy is always derived from `customer_line_description(item, line)`, and `public_description` is null and rejected by the API. For non-catalog lines, `public_description` is required when the line appears on a customer document.

## Open Decision

### Catalog Granularity

There are two viable shapes:

| Shape | Example | Pros | Cons |
|---|---|---|---|
| Style + color row, size on line item | `MORI-89216-IVORY`, line size `08` | Cleaner browsing, one public code per style/color, same dress keeps the same public code across sizes | Reorder flow must pass size separately |
| Style + color + size row | `MORI-89216-IVORY-08` | Matches many retailer ordering workflows, one catalog row maps to one exact orderable item | Catalog grows quickly, size 8 and size 10 get different public codes |

Recommendation: start with **style + color rows and size on the invoice/special-order line** unless retailer ordering proves that each exact size needs its own SKU row. This keeps customer codes stable for the same dress even if the ordered size changes.

If the shop needs exact vendor order SKUs later, add `catalog_item_variants` for size-level internal ordering without changing customer-facing public codes.

## Morilee Shopify Seed Mapping

Morilee's public Shopify `products.json` feed gives enough data for a v1 catalog seed. Use a one-pass scraper against the collection endpoint and write a single JSON file. Do not fetch metafields, do not trust public prices, and do not guess sizes.

Confirmed mapping from the live response:

- Shopify `vendor` maps to `designer` (`Morilee`).
- First non-empty variant `sku` maps to `style_number` (`4080000` for Beatriz).
- Each product becomes one catalog seed row per color. Colors come from `options[0].values`; Beatriz produces `Black/Red Rose` and `Regal Royal`.
- `internal_sku` is derived as `<vendor-code>-<style_number>-<color-slug>`, e.g. `MORI-4080000-BLACK-RED-ROSE`.
- `body_html` is stored as source HTML and stripped into plain `description_text`.
- `images[].src` is stored as `image_urls`; the CDN URLs are high enough resolution for catalog use.
- `product_type` derives `category`; `Quinceanera Dresses` maps to `quince_gown`.
- Shopify `price` is ignored because Morilee exposes placeholder pricing on the public storefront.
- Tags and sub-line details are ignored in v1 because they are absent or live in metafields outside `products.json`.
- Sizes are intentionally omitted. Morilee publishes styles and colorways; Bellas owns size lists per style.
- `public_code` remains null in the seed. The app mints it when rows are imported into the real catalog table.
- The script assumes Morilee's current variant shape where all variants on a product share the style-number SKU and color is the only meaningful public variant axis. A future vendor with per-size variant SKUs needs a different importer.
- `data/seeds/morilee_vizcaya.json` is committed as stable seed input data. It is small enough for git and gives the team reproducible catalog source data even if the public endpoint changes later.
- `description_html` stays in the seed for auditability, but Phase 1 import should drop it and persist only `description_text` unless a real admin use case for source HTML appears.

Default scraper location:

```bash
python scripts/seed_catalog/morilee.py
```

Default output:

```text
data/seeds/morilee_vizcaya.json
```

Default scope is the whole Vizcaya collection endpoint. If Bellas later needs only Valencia, Valentina, and Vizcaya sub-lines, that becomes either separate collection endpoint discovery or a later metafield pass. v1 intentionally seeds the full collection response.

## Multi-Vendor Seed Scrapers (shipped)

The Morilee scraper proved the seed pattern; six more vendors now follow it. Each vendor's site is different, so the *discovery* differs, but every scraper emits the same seed-JSON contract and the same import path applies (`scripts/seed_catalog/import_seed.py` mints `public_code` on insert). One catalog row per (style, color); `public_code` is null in the seed.

| Vendor | SKU prefix | Platform | Discovery |
|---|---|---|---|
| Morilee | `MORI` | Shopify `products.json` | collection feed + product-page enrichment |
| Ariana Vara | `ARIA` | Syvo storefront | category `?page=N` HTML + detail pages |
| Q by DaVinci | `QDV` | DaVinci async API | `async/loadmore` JSON + detail images |
| Rachel Allan | `RACH` | server-rendered HTML | per-color detail URLs |
| House of Wu | `WU` | WordPress REST API | `/wp-json/wp/v2/dress?dress_collection=<term>` |
| Layla K (May Queen) | `LAYLA` | Magento | category `?p=N` + grouped-product variant table |
| GLS Collective | `GLS` | WooCommerce Store API | `/wp-json/wc/store/v1/products?category=<id>` |

Seed contract consumed by the importer ΓåÆ `CatalogItemInput`: `internal_sku` (`VCODE-STYLE-COLORSLUG`), `color`, `category` (`quince_gown`/`bridal_gown`/`formal_gown`/`accessory`/`service`), `designer`, `style_number`, `house_name`, `product_title`, `description_text`, `image_urls[]`, and a `source` block ΓåÆ `{platform, product_id, handle, product_url, source_url, product_type}`. Extra fields (`color_hex`, `size_range`, `attributes`, `image_meta`, `source.price_cents`) ride along unused until a schema change consumes them. Re-import is create-missing by default; `--update-existing` applies diffs but only on the `REFRESH_ALLOWLIST` (display/image/source fields ΓÇö never identity columns).

Adding a vendor is purely a scraper + import; **no frontend or backend change**. Once rows land with a new `designer`, the Products page vendor chip appears automatically (see below).

Two load-bearing lessons baked into the scrapers:

- **Colors come from the authoritative variant list, never from photos.** Layla K originally derived colors from Magento media-gallery captions and produced phantom colors (photographed but not orderable) and junk labels; it was switched to the grouped-product table (the actual sellable variants) and audited against every other vendor. Ariana Vara (swatches), Morilee (Shopify options), House of Wu (`dress_color` taxonomy), Rachel Allan (per-color variant URLs), and Q by DaVinci (API `color` attribute) were all confirmed authoritative.
- **Wholesale/login-gated prices stay out of the customer price field.** GLS Collective's WooCommerce Store API leaks wholesale prices even though the public site hides them. They are captured in `source.price_cents` for reference only; `unit_price_cents` (which pre-fills customer quotes/invoices) is left null, consistent with every other vendor.

Seeds are committed under `data/seeds/` as reproducible source data, the same as `morilee_vizcaya.json`.

## Products Browse Page (shipped)

The `/products` admin page (`frontend/src/pages/AdminCatalog.jsx`) is the staff catalog surface. Beyond the operational table it adds, for browsing a multi-thousand-row catalog without endless scrolling:

- **Gallery / List view toggle.** Gallery shows image cards grouped by `designer + style_number` with color swatches; List is the operational per-row table with sortable, sticky headers.
- **Color-family filter.** A keyword classifier (whole-word matching) groups free-text color names into families (Pinks, Blues, Metallics, ΓÇª) so staff filter by vibe, not exact name. The map is editable in one place; a stored `color_family` is the eventual replacement.
- **Server-side Vendor filter.** Sourced from `GET /api/catalog/designers` (distinct designers + counts) so vendors past the per-request row cap still appear. The catalog exceeds the API's 500-row `limit`, so vendor is filtered server-side (each vendor fits under the cap on its own); color-family/sample/sort run client-side over the returned vendor set.

The **"Add-ons" bucket is labeled "Alterations"** across the admin page, the line-item picker, and the customer-facing line description (the `service` category label in `services/catalog_service.py`). The underlying `addon` UI bucket and `service` enum are unchanged ΓÇö it is a label-only rename, no migration.

## Public Code Format

Locked format: **`BVX-NNNNN`** with zero-padded 5-digit sequential numbering, drawn from a single global sequence regardless of vendor.

```text
BVX-00001
BVX-00002
BVX-00003
```

Vendor identity is intentionally not encoded in the public code. A customer holding `BVX-00042` cannot tell whether the dress is a Morilee, an Allure, or a Bellas-only sample. Vendor lives in the staff-facing `internal_sku` and the `designer` column, never in `public_code`.

Sequential is the v1 default because it reads cleanly over the phone and the active threat model is "customer Googles the designer SKU," which sequential already defeats. If Bellas later needs to hide sale order or volume from customers comparing two invoices, future codes can switch to short random tokens (e.g. `BVX-7K2P`) without regenerating existing codes.

Do not regenerate old codes. Once a public code appears on an invoice, it is part of the audit trail.

## Customer-Safe Rendering

The unsafe version:

```text
Mori Lee 89216 Ivory Size 08
```

The safer default:

```text
BVX-00001
Quince gown / Ivory / Size 08
```

The strongest obfuscation:

```text
BVX-00001
Isabella gown / Ivory / Size 08
```

The second version hides the designer/style number but still describes what the customer bought. The third version replaces designer identity with a Bellas house name. Pick one deliberately.

Customer-facing separators are forward slashes with surrounding spaces (` / `), never em dashes. The project's customer-copy rule forbids em dashes, so the separator choice is part of the contract, not a stylistic preference. Any formatter that ever renders to a PDF, portal page, email, SMS, or Stripe payload must use ` / `.

## Proposed Tables

### `catalog_items`

One row per orderable catalog item, probably style + color in v1.

- `id SERIAL PRIMARY KEY`
- `internal_sku VARCHAR(160) NOT NULL UNIQUE`
- `public_code VARCHAR(32) NOT NULL UNIQUE`
- `designer VARCHAR(120)`
- `style_number VARCHAR(80)`
- `color VARCHAR(80) NOT NULL`
- `house_name VARCHAR(120)`
- `product_title VARCHAR(200)` populated from the source title at import (e.g. `Beatriz Quincea├▒era Dress`); searchable so staff can type a dress name and find every color row
- `category VARCHAR(40) NOT NULL CHECK (category IN ('quince_gown', 'bridal_gown', 'formal_gown', 'accessory', 'service'))`
- `description_text TEXT`
- `image_urls JSONB NOT NULL DEFAULT '[]'::jsonb`
- `source_platform VARCHAR(40)`
- `source_product_id VARCHAR(80)`
- `source_product_handle VARCHAR(160)`
- `source_product_url TEXT`
- `source_collection_url TEXT`
- `source_product_type VARCHAR(120)`
- `is_sample BOOLEAN NOT NULL DEFAULT false`
- `active BOOLEAN NOT NULL DEFAULT true`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

Constraints:

- `category` is constrained by CHECK to a known set; new categories require a migration so freeform values cannot drift in.
- `public_code` is immutable in service code after insert.
- `internal_sku` is staff-visible only and is never returned from public endpoints.

`is_sample` is a flag, not a count. Floor samples are tracked as a boolean for v1; if the business later needs to track how many samples exist or where they live, that becomes its own small feature, not a counter on the catalog row.

### `invoice_line_items`

Extend the existing line-item table instead of creating a separate sales table.

- `catalog_item_id INTEGER NULL REFERENCES catalog_items(id) ON DELETE RESTRICT`
- `size_label VARCHAR(40)` for ordered size when catalog rows are style + color
- `public_description TEXT`
- `internal_notes TEXT`

Rules:

- New catalog-backed lines do not write to the legacy `description` or `notes` columns. Customer copy comes from `customer_line_description(item, line)`; `public_description` is null and unsettable on these lines (the API rejects writes to it). Staff context goes into `internal_notes`.
- New non-catalog lines write `public_description` (required when the line appears on customer documents) and optionally `internal_notes`. They do not write to the legacy `description` or `notes` columns.
- Legacy lines (no `catalog_item_id`) keep rendering their existing `description` as-is on customer surfaces, because that text is already on issued PDFs and portal pages. They do **not** keep rendering `notes` after the Phase 4 render swap: Phase 0 confirmed `notes` was never customer-safe by design, so it is dropped from public render for every line including legacy. The `notes` column stays in the schema for historic staff context.
- `internal_notes` is staff-only and never returned from public APIs, PDF contexts, portal endpoints, activity timeline payloads, or notification render contexts.
- A service-level guard rejects any write to a public-facing field whose value contains the matched catalog row's `internal_sku`, `designer`, or `style_number` as a case-insensitive substring. This guard ships in Phase 2, not Phase 7.

### `quote_line_items`

Apply the same column additions and the same write/render rules listed above for `invoice_line_items`:

- `catalog_item_id INTEGER NULL REFERENCES catalog_items(id) ON DELETE RESTRICT`
- `size_label VARCHAR(40)`
- `public_description TEXT`
- `internal_notes TEXT`

This repo's quotes have their own table, service, and editor, so the catalog migration touches both write paths. A quote that converts to an invoice must not lose its catalog linkage or revert to staff-typed customer copy.

### `special_orders`

Tracks "where is my dress" without pretending the shop has a stock-counting inventory engine.

- `id SERIAL PRIMARY KEY`
- `event_id INTEGER REFERENCES events(id) ON DELETE RESTRICT`
- `invoice_line_item_id INTEGER REFERENCES invoice_line_items(id) ON DELETE SET NULL`
- `catalog_item_id INTEGER REFERENCES catalog_items(id) ON DELETE RESTRICT`
- `size_label VARCHAR(40) NOT NULL`
- `status VARCHAR(24) NOT NULL DEFAULT 'ordered'`
- `ordered_at TIMESTAMPTZ`
- `eta_date DATE`
- `received_at TIMESTAMPTZ`
- `picked_up_at TIMESTAMPTZ`
- `vendor_order_number VARCHAR(120)`
- `internal_notes TEXT`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

Statuses:

- `needed`
- `ordered`
- `delayed`
- `received`
- `picked_up`
- `cancelled`

## Formatter Contract

Backend:

```py
def staff_sku(item: CatalogItem) -> str:
    return item.internal_sku


def customer_sku(item: CatalogItem) -> str:
    return item.public_code


def customer_line_description(item: CatalogItem, line: InvoiceLineItem) -> str:
    label = item.house_name or category_label(item.category)
    size = f" / Size {line.size_label}" if line.size_label else ""
    return f"{label} / {item.color}{size}"
```

Frontend:

```ts
export function staffSku(item: CatalogItem): string {
  return item.internal_sku
}

export function customerSku(item: CatalogItem): string {
  return item.public_code
}
```

Avoid:

```ts
displaySku(item, "customer")
displaySku(item, "staff")
```

One typo like `"customers"` can quietly leak the internal SKU if the fallback is wrong. Named helpers make the call site obvious and easier to search.

The customer formatter must use ` / ` as its separator. Em dashes are forbidden in customer copy across this project, so the separator choice is part of the formatter contract, not a stylistic preference. Any future contributor reaching for an em dash in `customer_line_description` is changing the contract, not just the copy.

The original plan focused on `description`; `notes` is the same leak class on the same render path. **Phase 2 must address both columns**, not just `description`.

`product_key VARCHAR(120)` exists on both tables but is **not currently rendered** to any customer template (only echoed by the staff `LineItemResponse` at [api/routers/invoices.py:116](api/routers/invoices.py#L116)). Safe today; the forbidden-fields list below keeps it that way.

Grandfather decision for legacy lines (no `catalog_item_id`): keep rendering `description` as-is on customer surfaces, because that exact text is already on issued PDFs in customers' inboxes and on portal pages they have bookmarked. **Do not grandfather `notes` rendering.** `notes` was never confirmed customer-safe by design; staff may have used it as private side notes that happen to render today. The Phase 4 render swap stops `notes` rendering on every line, including legacy. The migration only stops *new* lines from writing leakage; the render swap stops `notes` exposure for everyone.

#### Forbidden public response fields

For any public endpoint, public PDF context, public HTML render, public email body, future Stripe payload, and any activity payload that ever becomes customer-visible:

- Catalog row: `internal_sku`, `designer`, `style_number`, `internal_notes`
- Invoice/quote: `private_notes` ([database/models.py](database/models.py); not rendered today, must stay that way)
- Invoice installments: `staff_notes` ([database/models.py:482](database/models.py#L482))
- Payment: `notes`, `transaction_reference` ([templates/pdf/receipt.html:9, :71](templates/pdf/receipt.html#L9))
- Line items: `product_key`; `notes` and `description` for any catalog-backed line
- Activity payload: line-item `description`, `product_key`, or catalog `internal_sku`/`designer`/`style_number`
- Free-form staff fields: `quote.rejection_reason`, `quote.cancellation_reason`, `invoice.cancellation_reason`

#### Findings that affect later phases

1. **Line `notes` is also a leak vector.** Phase 2 schema and rules must split `notes` (rename to `internal_notes`, add `public_description`) on both `invoice_line_items` and `quote_line_items`. The original phase wording focused on `description`; both columns need the same treatment.
2. **No Stripe today.** Phase 4's conditional Stripe task converts to a forward-looking note: when Stripe lands later, route through `customer_sku` and `customer_line_description` from day one.
3. **PDF cache is essentially empty (1 file across 23 directories).** Phase 4's regenerate-vs-grandfather decision becomes "wipe cache directories, let regeneration happen on next download."
4. **Receipt PDF leak path is dormant.** No customer-facing route serves receipts today, but `templates/pdf/receipt.html` already renders `payment.notes` and `payment.transaction_reference`. Any Phase 4 work that adds a customer receipt route must first split or strip those fields.

## Acceptance Criteria

- Staff invoice builder shows `internal_sku`.
- Customer invoice PDF shows `public_code`.
- Customer portal pages, activity timeline entries, reminder emails, dunning emails, AR rollup customer-facing outputs, payment receipts, and Stripe-bound payloads never expose `internal_sku`, `designer`, `style_number`, `internal_notes`, `product_key`, or the legacy `notes` column.
- Catalog-backed invoice and quote lines do not render staff-entered free text to customers.
- Legacy invoice and quote lines (no `catalog_item_id`) keep rendering their original `description` on customer surfaces but stop rendering `notes` after the Phase 4 render swap.
- Non-catalog invoice lines can still represent deposits, rush fees, alterations, and discounts via `public_description`.
- A test fixture containing `Mori Lee 89216 Ivory 8` in internal fields does not leak to any public surface (PDF, portal, timeline, email, SMS, Stripe).
- The forbidden-substring guard rejects writes that try to put `internal_sku`, `designer`, or `style_number` into a public-facing field.
- Customer-facing copy uses ` / ` separators, never em dashes.
- Special orders can be tracked through ordered, ETA, received, and picked up states.
